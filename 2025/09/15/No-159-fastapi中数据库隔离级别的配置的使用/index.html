No.159-fastapi中数据库隔离级别的配置的使用

Hope
2025-09-15 (Updated: 2025-09-15)
fastapi

导读

在做一次数据库插入数据的时候，发现重复插入问题，经过排查发现是并发导致。于是开始排查原因。

问题原因排查经过

采用分布式锁

使用了分布式锁对操作数据库逻辑进行了加锁，但是发现还是会有重复插入问题。

发现每次插入数据前都会查询一次数据，如果没有就插入，有就跳过。

添加调试日志

加了锁还是不能解决问题。于是将操作数据库前后的日志打印出来。

发现当两个请求同时查询和插入的时候，明明已经加锁了，但是数据插入没成功，虽然每次请求都显式进行了 commit 操作。

所以结果就是，两个请求都查询了数据库，发现没有数据，然后都插入了数据，导致重复插入。

解决方案

经过一番搜索，发现：如果你用的数据库（比如 MySQL 默认 InnoDB），事务隔离级别是 REPEATABLE READ。
线程 A 插入并提交，线程 B 在插入前就开启了事务，那么它的快照还是旧的，导致查不到最新提交的数据。

所以，解决方案就是：将数据库隔离级别改为 READ COMMITTED。

具体 fastapi 中操作

在创建数据库引擎的时候，添加 isolation_level="READ COMMITTED" 参数即可。

再使用依赖注入的方式，统一获取 Session 对象，这样在跨 Session 的情况下，就可以避免重复插入了。

总结

数据库隔离级别，这个参数在平时开发中，很少用到，但是当遇到并发问题的时候，这个参数就派上用场了。

遇到问题，不要慌，先排查原因，可以使用 debug 模式，打印日志，一步步排查。

MySQL 支持以下隔离级别（SQLAlchemy 里字符串要写全大写）：

  • “READ UNCOMMITTED”
  • “READ COMMITTED” ✅ 推荐，能看到别的事务已提交的数据
  • “REPEATABLE READ” （MySQL 默认，可能导致你查不到刚提交的数据）
  • “SERIALIZABLE”
