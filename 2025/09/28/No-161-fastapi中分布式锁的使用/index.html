🚀 FastAPI 中分布式锁的使用：从一次“库存超卖”说起

去年双 11，我接到一个临时需求：某个电商项目要在 FastAPI 微服务中接入一个限时秒杀活动。听起来很简单：下单时，先扣减库存，再生成订单。

但活动刚开始 5 分钟，测试环境里就出现了库存被超卖的问题。

我一脸问号：

“不是加了数据库行级锁吗？怎么还会超卖？”

后来排查发现：

  • FastAPI 部署在 多实例 + 多进程 模式下。
  • 请求并发打到不同节点，单机锁根本没法控制全局并发。
  • 导致库存还没刷新，另一个节点也以为还能买。

这时，我们才意识到：
👉 需要一把分布式锁。

🔑 为什么要分布式锁？

常见场景：

  • 秒杀/抢购：防止库存被多减
  • 任务调度：避免多个副本同时跑同一个任务
  • 接口幂等：防止重复请求造成数据重复写入

在 单机环境 下，threading.Lock 就够了。
但在 分布式环境（Kubernetes / 多副本部署）下，必须要靠 Redis 这种 中心化存储 来协调。

🧩 分布式锁的原理

分布式锁的基本原理很简单：

  1. 加锁：尝试在 Redis 设置一个 Key（只允许第一个写入成功）。
  2. 执行业务逻辑：执行库存扣减、报表生成等。
  3. 解锁：执行完后删除 Key，但要注意不能误删别人的锁。

Redis 的 SETNX + 过期时间（EXPIRE）就能实现这一过程。

🧪 多副本 + 多线程脚本场景

分布式锁不仅能保护接口，还能用在 后台定时任务。

比如：多个 FastAPI 副本同时启动了一个定时脚本去生成报表，如果没加锁，结果就是——每个副本都生成一次报表，浪费资源还可能产生重复数据。

这也适用于多副本的消费者脚本场景：例如高并发情况下，多个消费者同时向数据库插入数据，虽然插入之前都会进行检查数据是否存在，但还是会出现重复数据。这个时候就需要用到分布式锁来保证数据的唯一性。

即使部署了多个副本，也只有一个实例能拿到锁执行，避免重复任务。

⚠️ 踩坑与优化

  1. 锁过期时间太短

    • 业务没执行完，锁就过期，其他实例进来重复执行。
    • ✅ 解决：加“锁续期”（看门狗）。

  2. 锁过期时间太长

    • 服务宕机后锁迟迟不释放，影响可用性。

  3. 高并发下锁冲突严重

    • 可以结合 消息队列 来削峰填谷。

🏁 总结

那次“库存超卖”事故后，我对分布式锁有了更深的认识：

  • 在 单机时代，threading.Lock 足够。
  • 在 分布式环境下，Redis 分布式锁是最常用的解决方案。
  • 在 FastAPI 项目里，配合 @contextmanager 写法，能让锁的获取与释放更安全、优雅。

一句话：

分布式锁不是银弹，但在关键场景下，它是救命稻草。

每日踩一坑，生活更轻松。
