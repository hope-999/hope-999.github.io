No.162-fastapi中使用 celery.backend_cleanup 的工作原理

Hope
2025-10-31 (Updated: 2025-10-31)
Celery

导读

FastApi 中使用定时任务和异步任务，首选 Celery。
在启动 Celery Beat 的时候，会自动初始化一个 celery.backend_cleanup 任务。
那么这个任务做了哪些事情呢？工作原理是什么？怎么使用和配置呢？

作用

它的作用：

  • 遍历 result backend；
  • 找出超过 result_expires 时间的任务结果；
  • 将其删除。

关键配置项：

配置项 说明 示例
result_expires 任务结果保存的过期时间（秒） app.conf.result_expires = 3600（1小时）
result_backend 存储任务结果的后端 redis://localhost:6379/0
result_backend_cleanup_interval（Celery 5.3+） 清理任务运行间隔 app.conf.result_backend_cleanup_interval = 3600

如果你用的 Beat scheduler 为数据库，则会创建一条定时任务数据，通常是 0 4 * * *，表示在每天的凌晨 4 点执行。

注意事项：

  • 如果你关闭了 result_persistent=False 或根本没用 backend（比如只 fire-and-forget），则无需关心。
  • 如果你用的是 数据库 作为 backend（如 Django ORM、SQLAlchemy），清理是删除数据库记录；
  • 如果你用的是 Redis，则清理是删除对应的 key；
  • 对于自定义 backend，必须实现 cleanup() 方法，否则该任务无效。

工作原理

  1. 首先判断结果数据是否过期
  2. 再根据筛选出来的过期数据，进行删除

判断过期数据的逻辑：Celery 通过比较 任务结果的存储时间 + result_expires 来判断是否过期。

不同 Backend 的判断方式：

Backend 类型 过期判断方式 说明
Redis / Memcached 利用 key 的 TTL（过期时间） Celery 在写入结果时会设置 EXPIRE，Redis 自动清除
Database（如 Django ORM、SQLAlchemy） 读取 date_done 字段，对比当前时间 Celery 定期执行 DELETE FROM celery_taskmeta WHERE date_done < now() - result_expires
RPC / AMQP 不持久化结果或自动过期 一般不需清理
Filesystem / S3 等持久化存储 检查保存文件的时间戳 比对 mtime 或 metadata
Celery.backend_cleanup 自动清理过期数据 Celery 定期执行 backend_cleanup，清理过期数据

过期数据常见的存储方式：

Backend 存储位置 示例
Redis Redis 的 key-value key 类似于 celery-task-meta-<uuid>
Database（Django ORM / SQLAlchemy） 数据库表中（默认表名：celery_taskmeta） 列：id, task_id, status, result, date_done, traceback
AMQP (RabbitMQ) 消息队列，不持久化 任务完成后即消费
Filesystem 文件系统的结果文件 每个任务结果单独存文件
Cache backend (memcached) 缓存中 自动过期
S3 / HTTP 远程对象存储 带 metadata 标识时间戳
