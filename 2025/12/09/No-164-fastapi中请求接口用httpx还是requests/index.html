No.164-fastapi中请求接口用httpx还是requests

Hope
2025-12-09 (Updated: 2025-12-09)
fastapi, httpx, requests

导读

在 FastAPI 项目中，我们经常需要调用外部 API 接口。Python 生态中有两个主流的 HTTP 客户端库：requests 和 httpx。

requests 是 Python 中最流行的同步 HTTP 库，而 httpx 是一个现代化的 HTTP 客户端，同时支持同步和异步操作。

那么在 FastAPI 这种异步框架中，我们应该选择哪个呢？本文将从技术特性、性能、兼容性等多个维度进行深入分析。

httpx vs requests：核心区别

1. 异步支持

httpx：

  • 原生支持异步操作，基于 asyncio
  • 可以无缝集成到 FastAPI 的异步路由中
  • 支持 async/await 语法

requests：

  • 仅支持同步操作
  • 在异步环境中使用会阻塞事件循环
  • 需要使用线程池或 asyncio.to_thread() 包装

2. HTTP/2 支持

httpx：

  • 原生支持 HTTP/2 协议
  • 可以提升性能，特别是在需要多次请求的场景下

requests：

  • 仅支持 HTTP/1.1
  • 不支持 HTTP/2

3. 性能对比

特性 httpx requests
异步支持 ✅ 原生支持 ❌ 需要包装
HTTP/2 ✅ 支持 ❌ 不支持
连接池 ✅ 自动管理 ✅ 支持
超时控制 ✅ 更灵活 ✅ 支持
流式响应 ✅ 支持 ✅ 支持
并发性能 ✅ 优秀 ⚠️ 受限于 GIL

4. API 设计

httpx：

  • API 设计与 requests 高度相似，学习成本低
  • 支持同步和异步两种模式
  • 更现代的 Python 特性支持

requests：

  • 成熟稳定，API 简洁直观
  • 生态丰富，文档完善
  • 仅支持同步模式

7. 依赖和兼容性

httpx：

  • 依赖较少，但需要 Python 3.8+
  • 与 FastAPI、Starlette 等异步框架完美集成
  • 可以替代 requests 的大部分功能

requests：

  • 依赖 urllib3，生态成熟
  • Python 2.7+ 和 Python 3.x 都支持
  • 在同步代码中表现优秀

总结：什么时候用哪个？

推荐使用 httpx 的场景

  1. FastAPI 异步路由中调用外部 API（强烈推荐）

    • 原生异步支持，不会阻塞事件循环
    • 性能更好，可以充分利用异步并发

  2. 需要 HTTP/2 支持

    • 某些现代 API 服务要求或推荐使用 HTTP/2

  3. 高并发场景

    • 需要同时发起大量 HTTP 请求
    • 异步特性可以显著提升性能

  4. 需要 WebSocket 支持

    • httpx 可以处理 WebSocket 连接

可以使用 requests 的场景

  1. 同步代码环境

    • 不在异步框架中，使用 requests 更简单直接
    • 注意：httpx 也支持同步模式（httpx.Client），API 与 requests 几乎一致，可以作为 requests 的现代化替代

  2. 已有大量 requests 代码

    • 迁移成本较高，可以继续使用
    • 但建议逐步迁移到 httpx（同步模式迁移成本很低）

  3. 简单的脚本或工具

    • 不需要异步特性，requests 足够使用
    • 也可以考虑使用 httpx 的同步模式，获得更好的性能和 HTTP/2 支持

在 FastAPI 中的最佳实践

注意事项：

  • 使用全局客户端可以复用连接，提升性能
  • 记得在应用关闭时调用 aclose() 关闭客户端
  • 合理设置超时和连接限制，避免资源耗尽

结论

在 FastAPI 项目中，强烈推荐使用 httpx，原因如下：

  1. 原生异步支持：与 FastAPI 的异步特性完美匹配，不会阻塞事件循环
  2. 更好的性能：在高并发场景下，异步特性可以显著提升性能
  3. 现代化特性：支持 HTTP/2、WebSocket 等现代协议
  4. API 兼容：与 requests API 高度相似，迁移成本低
  5. 官方推荐：FastAPI 官方文档中也推荐使用 httpx 进行异步 HTTP 请求

虽然 requests 仍然是一个优秀的库，但在异步框架中使用它会带来性能损失和代码复杂度。除非有特殊原因（如需要兼容旧代码），否则在 FastAPI 中应该优先选择 httpx。
